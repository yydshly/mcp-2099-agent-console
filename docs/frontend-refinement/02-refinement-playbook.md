# 交互前端打磨方法论

## 适用范围

适用于从提示词、参考图、低保真原型或已有代码出发，交付高质感、可操作、可验证的交互前端，例如产品页、数据大屏、运营后台、工作台、指挥台和沉浸式 WebGL 页面。

不依赖某个业务领域、Agent 协议或后端实现。

## 核心定义

前端打磨不是持续加动画或修改颜色，而是持续缩小以下四类差距：

```text
设计意图 ↔ 浏览器实际画面
信息层级 ↔ 用户可读性
操作预期 ↔ 交互反馈
代码状态 ↔ 交付可靠性
```

只有先看到真实差距、明确分类、做最小修复并检查相邻状态，打磨才会收敛。

## 工作流

### 1. 将输入拆成可验证约束

从提示词或参考图中提取，不直接照着写页面：

- **视觉约束**：色彩、材质、密度、空间、动效、摄影感、字体气质。
- **信息约束**：首屏主次、必须持续可见的信息、辅助信息、空状态。
- **操作约束**：用户可以打开什么、填写什么、确认什么、看到什么反馈。
- **状态约束**：空闲、加载、进行、成功、暂停、失败、取消、不可用。
- **环境约束**：桌面/移动、亮/暗主题、低性能、无 WebGL、键盘和语言。

输出一页“视觉层 / 信息层 / 操作层 / 状态层”清单。若无法写清操作闭环，先不要投入视觉细化。

### 2. 建立可运行的最小基线

先保证浏览器可访问，再追求效果：

1. 页面根布局覆盖目标视口。
2. 主视觉区域、信息区域、操作入口同时存在。
3. 主视觉是可替换增强层；没有它时页面仍可阅读和操作。
4. 首个核心操作可以完成最小闭环。

不要在静态截图状态下连续堆叠样式。每个阶段都必须在真实浏览器中观察。

### 3. 用浏览器证据驱动迭代

每轮只记录一个用户可观察的问题：

```text
现象：按钮在亮色主题中看不见
分类：可读性 / 主题 token
根因：暗色低透明文字被直接复用
最小修复：补齐亮色语义色，不改变按钮状态逻辑
回归面：表单、抽屉、禁用、悬停、焦点、移动端
完成条件：同一语义控件在全部主题可读
```

优先使用截图、DOM、实际操作和构建错误作为证据。不要根据代码想象页面结果。

### 4. 按问题类别选择干预层

| 问题 | 优先检查 | 不应先做的事 |
| --- | --- | --- |
| 主视觉不像参考 | 构图、材质、光照、密度、镜头、运动 | 只改背景色 |
| 字体/控件看不见 | 主题 token、语义色、禁用/空态 | 局部强行加白色 |
| 面板重叠 | 前景层状态机、布局边界、关闭路径 | 无限提高 z-index |
| 操作无反馈 | 异步状态、通知、禁用、错误路径 | 只添加 console log |
| 状态互相矛盾 | 单一显示映射、状态源 | 每个组件各自翻译枚举 |
| 手机布局断裂 | 信息优先级、动态内容出现后的空间 | 只缩小字号 |
| Tab/Escape 异常 | 焦点范围、前景层、关闭优先级 | 只拦截按键 |
| WebGL 卡顿/不可用 | 质量档、能力判断、懒加载、回退 | 强制降低所有用户效果 |
| 文案遗漏 | 状态、空态、导出、fallback、aria | 只翻译导航 |

### 5. 前景层与操作规则

为抽屉、弹层、命令面板、详情卡和确认门定义明确规则：

- 同时只保留一个主要前景操作层；确认门可覆盖其触发层。
- 每个可关闭层都有可见关闭入口。
- `Escape` 从最上层开始关闭，不做无条件全量重置。
- 打开层后将焦点移入；关闭后回到触发元素。
- 破坏性动作进入确认流；成功和失败都有可见反馈。
- 表单显示值、校验值和提交值使用同一数据解析路径。

### 6. 横向回归矩阵

完成一次局部修复后，至少检查受影响维度：

| 维度 | 最小检查项 |
| --- | --- |
| 主题 | 深色、亮色、切换中的可读性 |
| 尺寸 | 桌面、平板、手机；动态面板出现后的布局 |
| 状态 | 空闲、加载、成功、失败、暂停、取消、空状态 |
| 输入 | 默认值、空值、错误、禁用、键盘提交 |
| 浮层 | 打开、关闭、叠加、Escape、焦点返回 |
| 语言 | 主文案、状态、空态、导出、回退、aria |
| 性能 | 低性能、减少动效、无能力回退 |

不需要每次全量测试；根据修改类别选择最相关的横向面。准备交付时再做全量矩阵。

### 7. 高成本视觉的产品化规则

当页面包含 WebGL、Canvas、视频、大型滤镜或复杂动效：

- 将视觉运行时与业务控制层解耦。
- 延迟加载高成本模块。
- 提供自动质量策略和手动质量档。
- 检测能力、节能偏好、网络与减少动效偏好。
- 无能力时显示语义明确的静态回退。
- 主题转场分别审视两个方向，不假设亮转暗与暗转亮等价。

### 8. 交付收口

页面“能展示”不是交付完成。至少完成：

1. 真实浏览器可访问。
2. 构建、lint、测试和契约/接口校验通过。
3. README 描述真实项目，而不是脚手架模板。
4. 记录已完成、明确遗留和触发条件。
5. Git 不跟踪构建物、日志、依赖目录和重复源码备份。
6. 建立已验证的检查点提交。

## 停止规则

停止并记录，而不是继续修改，当满足以下条件：

- 当前用户旅程可完成。
- 已知问题没有明确视觉证据、验收要求或性能数据。
- 下一项修改无法说明会改善哪个具体用户可观察结果。
- 预计收益低于引入主题、响应式或状态回归的风险。

将此类事项写入待办，并记录触发条件，例如“收到真实设备帧率数据”“业务方确认新的视觉方向”“后端提供失败语义”。

## 常见反模式

- 根据代码而非浏览器画面判断效果。
- 用更多 `z-index` 处理没有状态机的浮层。
- 用局部颜色补丁替代主题语义色。
- 将内部枚举、字段名或服务端原始值直接显示给用户。
- 为了视觉效果跳过键盘、移动端、空态和异常态。
- 因为“还能优化”持续打扰已稳定基线。
- 未经构建和测试验证就提交所谓完成版本。

## 与案例的关系

任何项目都可以有自己的视觉语言、业务对象和技术栈。本方法论不复制这些内容；它只约束从“意图”到“可验证交付”的前端打磨路径。

MCP 2099 是一个包含 WebGL、浮层、主题、状态面和交付治理的案例，详细证据见 [01-iteration-evidence-log.md](01-iteration-evidence-log.md)。
