# 通用交互前端打磨 Skill 设计

## 定位

拟定名称：`interactive-frontend-refinement`

该 Skill 用于将提示词、参考图、原型或已有前端，逐步打磨为高质感、可操作、可验证、可交付的交互前端。

它不定义业务模型、后端协议、Agent 调度或领域流程。它可以用于产品展示页、运营工作台、数据大屏、复杂表单界面、沉浸式 WebGL 页面和 Agent Console。

## 触发条件

应在以下请求中触发：

- “根据这张参考图做一个可运行前端。”
- “把这个视觉原型打磨到可交付。”
- “页面效果不够好，系统排查并优化。”
- “这个 WebGL/Three.js 页面需要补齐控制台交互、主题和响应式。”
- “检查截图中控件遮挡、字体不可见、面板关闭、移动端布局等问题。”
- “从提示词创建电影级、互动式、高质感前端。”

不应仅因“实现一个普通 CRUD 页面”或“设计后端业务 API”而触发。

## 输入

Skill 应接受任意组合：

- 文字提示词。
- 参考图、录屏或现有页面 URL。
- 当前代码仓库与运行方式。
- 已知视觉/交互问题截图。
- 目标设备、主题、性能或无障碍约束。

## 输出

Skill 的输出不是固定页面，而是一个可迭代的交付过程：

1. 视觉/信息/操作/状态约束摘要。
2. 可运行的前端基线。
3. 基于真实浏览器证据的问题列表与最小修复。
4. 主题、响应式、状态、键盘、回退和本地化的横向验收。
5. README、待办、版本库卫生和验证结果。

## 核心流程

### 阶段 A：建立约束

从输入中识别视觉语言、信息层级、用户操作和状态面。先确定完成定义，再决定是否需要 WebGL、动画、抽屉或复杂前景层。

### 阶段 B：建立可运行基线

优先保证真实浏览器访问、根布局、主视觉、信息区和最小操作闭环。高成本视觉必须可降级。

### 阶段 C：浏览器驱动打磨

每轮以用户可观察的问题为单位：收集截图/DOM/交互证据，分类根因，修改最小范围，检查相邻主题、尺寸和状态。

### 阶段 D：交互与可靠性收口

处理浮层层级、关闭、焦点、Escape、表单状态、成功/失败反馈、空态、键盘、主题、响应式、语言和能力回退。

### 阶段 E：交付收口

执行构建、lint、测试和项目特定验证；更新 README；清理 Git 噪音；记录不应继续做的待办及其触发条件。

## 必须包含的规则

- 浏览器实际观察优先于代码想象。
- 每次只解决一类确定问题，避免无边界重构。
- 前景层必须有明确关闭、Escape 和焦点规则。
- 状态显示通过统一映射层输出，不直接暴露内部枚举。
- 深浅主题验收覆盖所有状态面，不仅覆盖首页。
- 复杂视觉提供低性能、减少动效和不可用回退。
- 无明确验收或证据的“还能优化”进入待办，不继续实现。

## 与现有临时 Skill 的关系

现有 `agent-console-frontend-delivery` 是本项目早期的临时草稿，范围过窄。

最终实施时：

1. 创建新的 `interactive-frontend-refinement` 全局 Skill。
2. 将其中可复用的前端规则迁移到新 Skill。
3. 将 MCP 2099/Agent Console 保留为案例，不进入核心指令。
4. 旧 Skill 不再作为通用入口；可保留为 Agent Console 专项参考，或在确认后移除。

## 验证方式

新 Skill 创建后，应使用一个与 MCP 2099 不同的任务做前向验证，例如：

- 高信息密度数据工作台。
- 有亮暗主题和复杂表单的运营页面。
- 有 Canvas/WebGL 增强层的产品展示页。

验证目标不是复刻 MCP 2099，而是确认该 Skill 仍能指导“约束拆解、浏览器证据、最小修复、横向回归和交付收口”。
