# 前端迭代证据日志：从提示词到可交付版本

## 记录目的与边界

本日志还原 MCP 2099 前端从最初视觉提示到可交付基线的真实迭代过程。它关注提示词理解、视觉实现、浏览器观察、问题诊断、最小修复、回归控制和交付收口。

不将客服、运营、Agent 调度、后端协议设计视为通用前端方法；它们只在影响前端展示和交互时被提及。

## 证据模板

每项记录使用以下字段：

- **用户信号**：用户观察、截图或明确期待。
- **现象**：浏览器中的真实表现。
- **判断**：问题属于视觉、布局、交互、状态、性能、可读性或工程链路的哪一层。
- **干预**：采用的最小修改方向。
- **回归风险**：修改后最可能影响的相邻区域。
- **结果**：用户可观察的结果或验证结论。
- **沉淀**：`可复用前端规则` 或 `项目特例`。

## 阶段 0：从提示词建立视觉与交互基线

### 0.1 视觉目标不是普通后台

- **用户信号**：要求电影级、可交互、可直接运行的复古未来主义界面，并给出高密度红色神经核心参考效果。
- **现象**：普通卡片布局或静态背景无法承载“核心、粒子、轨道、遥测、控制台”的第一印象。
- **判断**：这是信息层级和主视觉共同决定的产品前台，不是给既有后台换皮。
- **干预**：确定三层基线：WebGL 感知核心作为环境层；左右遥测/说明作为信息层；顶栏、面板、任务入口作为操作层。
- **回归风险**：场景过强会遮挡信息，控制台过强又会削弱沉浸感。
- **结果**：后续所有改动均以“场景服务操作，而非操作服务场景”为判断标准。
- **沉淀**：`可复用前端规则`。先将提示词拆为视觉层、信息层、操作层和状态层，再开始写组件。

### 0.2 必须先获得可运行页面

- **用户信号**：多次要求“启动”，并反馈 `127.0.0.1`、`localhost`、`::1` 访问问题。
- **现象**：即使设计代码存在，只要开发服务不可访问，用户无法给出有效视觉反馈。
- **判断**：运行环境是设计迭代的前置条件；“代码已写”不等于“用户可评审”。
- **干预**：以 Vite 本地服务为基线，优先保证本机可访问，再进行视觉调试。
- **回归风险**：只在一个主机名或一个浏览器环境检查，会掩盖访问与 IPv4/IPv6 差异。
- **结果**：浏览器成为后续截图、交互和响应式验证的主证据来源。
- **沉淀**：`可复用前端规则`。任何高视觉项目的第一验收项都是“真实浏览器可访问”。

## 阶段 1：参考图差距与主视觉收敛

### 1.1 从“有一个球”到“有能量密度的核心”

- **用户信号**：指出其他作品拥有更密集、尖锐、发光的红色核心、网格球和空间碎片，当前效果差距明显。
- **现象**：初版三维物体可能具备形状，却缺少材质、脉冲、粒子密度、空间层次和摄影感。
- **判断**：问题不只是模型复杂度，而是发光、扰动、外围网格、碎片和构图共同形成的感知密度。
- **干预**：围绕核心材质、线框球、粒子场、碎片、镜头与颜色层级逐步增强，而不是把所有信息压到 CSS 背景中。
- **回归风险**：GPU 开销、亮色主题对比度、核心遮挡左右信息和低性能设备卡顿。
- **结果**：形成“WebGL 只负责氛围和状态感知，控制面板保持独立可读”的实现边界。
- **沉淀**：`可复用前端规则`。参考图对标时，按构图、材质、光照、运动、景深、信息密度逐项拆解，不能只复制颜色。

### 1.2 主视觉必须有降级策略

- **用户信号**：询问 WebGL 是否资源过大、是否可以优化，以及资源不足时的处理方式。
- **现象**：三维场景是体验核心，但并非每台设备都有相同 GPU、网络和 WebGL 支持。
- **判断**：性能不是最后压缩包体时才处理的技术问题，而是产品的可用性边界。
- **干预**：引入自动/节能/影院质量模式、设备能力判断、懒加载场景运行时，以及 WebGL 不可用的文字回退。
- **回归风险**：低性能设备不应因为显示“影院”而实际走高成本渲染；回退文案也必须随语言切换。
- **结果**：构建保留较大的懒加载 WebGL chunk，但常规控制台不被其阻塞。
- **沉淀**：`可复用前端规则`。高成本视觉能力必须同时提供质量档、自动策略和无能力回退。

## 阶段 2：从视觉样机到可操作前台

### 2.1 “入口在哪里”暴露了样机边界

- **用户信号**：持续追问任务下发入口、驱动方式，以及是否只是展示 Agent 场景。
- **现象**：主屏很有氛围，但用户没有清晰路径将输入变成可观察状态。
- **判断**：视觉首页不是业务前台；至少需要一个完整的“打开、填写、确认、执行、观察、结束”闭环。
- **干预**：加入新建任务、任务配置、审批、任务日志、队列、历史、结果、协议和日志工作区。
- **回归风险**：每加一个入口都可能制造面板遮挡、重复状态源或关闭路径缺失。
- **结果**：页面从“神经核心展示”升级为“有控制面和反馈面的交互前台”。
- **沉淀**：`可复用前端规则`。任何视觉原型进入产品阶段前，先补最小用户闭环，不要先扩展视觉细节。

### 2.2 业务场景只作为演示，不决定前端结构

- **用户信号**：反复确认客服、退款、运营是否被写死，以及多 Agent 如何展示。
- **现象**：具体示例有助于演示，但若让页面条件分支围绕某业务增长，后续无法复用。
- **判断**：这是前端架构边界问题，而非视觉问题。
- **干预**：以 Profile、输入字段、结果扩展和 Local Mock 场景承载示例；通用页面只展示任务生命周期和配置化字段。
- **回归风险**：动态字段直接显示内部枚举或英文键名，会破坏可读性和国际化。
- **结果**：Agent 场景成为前端演示实例，而不是视觉实现的硬编码前提。
- **沉淀**：`项目特例`。此项目使用 Agent 作为实例；通用前端方法只保留“配置化内容不要污染壳层”的规则。

## 阶段 3：面板、覆盖与关闭路径的反复打磨

### 3.1 下拉框与表单控件不能使用默认浏览器观感

- **用户信号**：指出新建任务中的下拉框白底、选项样式和整体控制台风格不一致。
- **现象**：原生控件或半定制控件在暗色场景中会突兀，在亮色场景中又可能缺少边界和文字对比。
- **判断**：表单控件是业务前台的高频交互区，不能只优化主视觉。
- **干预**：统一选择器、选项、边框、悬停、禁用、亮色覆盖与字段说明的视觉系统。
- **回归风险**：亮色模式、键盘焦点、下拉展开层级和移动端宽度。
- **结果**：表单成为控制台的一部分，而不再像浏览器默认 UI。
- **沉淀**：`可复用前端规则`。先为输入、选择、按钮、复选框、错误和禁用状态定义同一套控制面 token。

### 3.2 浮层覆盖是系统问题，不是单个 z-index 问题

- **用户信号**：多次发现新建任务与 Panels、Mission Trace、结果卡、队列、历史互相遮挡，且某些面板无法关闭。
- **现象**：每个面板单独使用绝对定位时，局部修复会让另一个入口重新压住标题、关闭按钮或内容区。
- **判断**：根因是缺少前景层规则，而不是某个数字不够大。
- **干预**：建立前景层互斥逻辑：打开新面板先关闭非必要前景层；确认门高于抽屉；任务轨迹和结果卡在前景层打开时退让；每个面板提供明确关闭入口。
- **回归风险**：关闭一个面板时误清空其他状态；移动端仍可能因固定位置重新重叠。
- **结果**：Panels、任务表单、队列、历史、结果和工作区拥有更稳定的显示优先级。
- **沉淀**：`可复用前端规则`。先定义“前景层状态机”和关闭优先级，再微调 `z-index`。

### 3.3 关闭入口必须是可见且可到达的

- **用户信号**：反复询问两个标签页为什么没有关闭按钮，指出 Close 被 Panels 挤出控件。
- **现象**：功能存在不等于可发现；小空间下标题、状态、按钮容易争夺同一行。
- **判断**：关闭不是装饰，是浮层的必备操作。
- **干预**：为任务轨迹、结果、历史、队列、任务表单、工作区和确认门提供可见 Close；压缩标题布局，避免关闭按钮被覆盖或溢出。
- **回归风险**：亮色模式下 Close 文字对比不足；移动端按钮变得过小。
- **结果**：用户可以从当前上下文关闭，而无需点击主页“逃离”面板。
- **沉淀**：`可复用前端规则`。任何非永久侧栏/弹层都必须有显式关闭动作和键盘退出语义。

## 阶段 4：昼夜主题与可读性不是简单换背景

### 4.1 亮色模式暴露了被暗色掩盖的问题

- **用户信号**：指出亮色场景下大量标签、按钮、表单、日志、审批和任务文字看不见。
- **现象**：暗色样式中的低透明白字、边框和禁用色，在浅背景上失去对比；局部覆盖又会遗漏抽屉和详情卡。
- **判断**：主题切换涉及完整语义色系统，不是背景色和一个太阳图标。
- **干预**：为亮色模式补齐文字层级、边框、表面、主按钮、禁用按钮、状态色、表单和工作区覆盖，并通过截图逐页排查。
- **回归风险**：亮色补丁过多造成 CSS 覆盖累积；橙色强调色过浅或过饱和。
- **结果**：亮色模式从“可切换但不可用”进入可阅读、可操作状态。
- **沉淀**：`可复用前端规则`。主题验收必须覆盖所有状态面，而非只检查首页标题和背景。

### 4.2 主题过渡要尊重方向差异

- **用户信号**：认为亮转暗尚可，暗转亮仍然生硬，期望内部粒子有适应性变化。
- **现象**：两个方向使用同一套渐变时，视觉亮度和粒子曝光曲线不对称。
- **判断**：主题动画不是布尔切换；亮度、核心能量、粒子密度和背景网格需要方向感。
- **干预**：引入主题转换方向、过渡时间、粒子/场景适应，并记录暗转亮仍属于后续视觉待办。
- **回归风险**：增加动画导致低性能设备卡顿或减少动效偏好被忽略。
- **结果**：主要切换体验改善，剩余的暗转亮精修被明确记录而非无限迭代。
- **沉淀**：`可复用前端规则`。双主题动画应分别评审两个方向，并设置“达到可用后停止”的门槛。

## 阶段 5：响应式与空间竞争

### 5.1 全屏不完整和边界暴露布局假设

- **用户信号**：指出页面右侧和下方有边框，未适配浏览器全屏。
- **现象**：固定尺寸、容器高度或视口计算不一致会让场景背景与页面根节点断开。
- **判断**：主视觉页面必须从根容器到场景容器都使用一致的视口策略。
- **干预**：统一页面、导航、状态栏、场景和抽屉的高度/宽度约束，避免白边和滚动残留。
- **回归风险**：修正桌面全屏后，移动端固定底栏或抽屉高度可能再次溢出。
- **结果**：页面形成连续的全视口舞台。
- **沉淀**：`可复用前端规则`。沉浸式页面先验证根布局、场景、导航和底栏四者的视口闭合。

### 5.2 左右信息与底部面板会竞争空间

- **用户信号**：发现任务运行后左下 Mission Trace 遮住 Node/Queue 信息；历史详情和列表也互相挤占。
- **现象**：绝对定位的卡片在状态变化后占据原本静态信息区。
- **判断**：响应式不是媒体查询末尾补一个宽度，而是动态信息密度的布局策略。
- **干预**：移动端将任务轨迹调整为底部安全区域；历史详情改为可控高度并让列表预留空间；根据前景层和视口调整信息块位置。
- **回归风险**：桌面修复不能直接复制到手机；详情收缩过度会损失操作按钮。
- **结果**：Node/Queue、任务轨迹、历史列表和详情能同时保持可见或可滚动。
- **沉淀**：`可复用前端规则`。对状态驱动面板，按“内容出现后”的最坏布局测试，而不是只测空页面。

### 5.3 移动端导航需要替代路径

- **用户信号**：窄屏下顶栏空间不足，工作区、日志、协议等入口不应消失。
- **现象**：直接隐藏导航可避免拥挤，但会使功能不可达。
- **判断**：响应式压缩必须保留信息架构，而非只保留当前页面。
- **干预**：在窄屏引入工作区菜单，保留界面、神经网络、协议和日志入口；菜单选择后自动收起。
- **回归风险**：菜单 Escape 与页面级 Escape 冲突，打开菜单后点击其他顶部控件可能遮挡。
- **结果**：移动端保留完整工作区路径，并补齐本地关闭规则。
- **沉淀**：`可复用前端规则`。小屏导航应从“隐藏”转为“重组入口”。

## 阶段 6：状态一致性与操作反馈

### 6.1 任务状态必须在所有表面一致

- **用户信号**：完成任务后仍出现 `LIVE`，完成节点没有全绿，结果已就绪但轨迹仍像执行中。
- **现象**：时间线、右侧遥测、结果卡、队列和底栏可能各自使用不同阶段或旧状态。
- **判断**：这是显示模型分散造成的状态映射问题，不是单个颜色错误。
- **干预**：统一任务阶段、显示状态、时间线标签、颜色、结果可见性和队列状态；完成时将最终节点显示为 Ready/完成语义。
- **回归风险**：暂停、取消、失败和审批等待不能被“完成态”覆盖；动态子 Agent 状态也需映射。
- **结果**：完成、失败、取消和运行中的文本/颜色趋于一致。
- **沉淀**：`可复用前端规则`。一个状态机必须有单一显示映射层，禁止每个组件自己翻译枚举。

### 6.2 默认表单值与提交状态必须同源

- **用户信号**：表单看似已填入默认值，但点击下发提示缺少必填项。
- **现象**：视觉层渲染了 Profile 默认值，提交校验只检查用户显式输入状态。
- **判断**：这是 UI 显示状态和命令构造状态不一致。
- **干预**：让字段显示、必填校验、提交载荷都通过同一个“当前值/默认值”解析函数。
- **回归风险**：重置草稿、切换 Profile、空字符串覆盖默认值和动态字段。
- **结果**：默认展示值可以直接参与审批和下发。
- **沉淀**：`可复用前端规则`。表单的显示值、校验值和提交值必须共享同一解析源。

### 6.3 操作必须有即时、非阻塞的反馈

- **用户信号**：用户无法确认暂停、恢复、队列排序、审批、导出是否已经发生。
- **现象**：部分操作只改变内部状态或审计记录，主屏没有即时反馈。
- **判断**：控制台操作需要可见确认，但不应每次都弹出阻塞模态框。
- **干预**：增加操作通知；为结果导出和历史导出提供下载开始提示；失败交由运行时状态和错误文案表达。
- **回归风险**：通知文本本地化遗漏、多个通知覆盖、成功提示早于真实 Promise 成功。
- **结果**：关键操作获得轻量确认，导出不再像“点了没反应”。
- **沉淀**：`可复用前端规则`。命令型 UI 为每个异步操作定义成功、失败和进行中的反馈策略。

## 阶段 7：键盘、焦点与浏览器边界

### 7.1 Tab 跳出应用不是“浏览器问题”

- **用户信号**：用户发现按 Tab 会跳出当前 UI，甚至打开 Mission Queue 后无法回到预期操作层。
- **现象**：浮层没有焦点边界，Tab 会继续遍历浏览器或被遮挡的背景控件。
- **判断**：这是面板可访问性和前景层管理问题。
- **干预**：为活跃前景层限定可聚焦元素；打开面板时进入首个控件，关闭后返回触发入口。
- **回归风险**：只做焦点陷阱会让焦点停留在背景；多个浮层同时存在时需要确定最上层。
- **结果**：键盘用户可以完成打开、操作、关闭、回到入口的闭环。
- **沉淀**：`可复用前端规则`。所有弹层/抽屉应设计焦点进入、焦点范围、焦点返回三件事。

### 7.2 Escape 需要优先级而非“关闭全部”

- **用户信号**：询问 Escape 的含义，并发现它和菜单、任务面板、队列等交互不一致。
- **现象**：全局 Escape 若直接清空全部状态，会误关工作区；若只关一个菜单，又可能留下确认门。
- **判断**：Escape 是层级命令，需要按最前景操作层处理。
- **干预**：定义关闭优先级：确认操作、命令、表单、历史、队列、日志、面板索引、工作区；审批确认保留显式批准/拒绝语义。
- **回归风险**：顶栏局部菜单和页面全局监听会竞争同一键盘事件。
- **结果**：Escape 的行为与视觉层级一致，不再像随机重置页面。
- **沉淀**：`可复用前端规则`。键盘全局快捷键必须由前景层状态机仲裁。

---

## 阶段 8：本地化与展示文本扫漏

### 8.1 文案切换不能只覆盖导航

- **用户信号**：指出 Auto/Eco/Cinema、状态码、空状态、日志、导出内容、WebGL 回退仍有英文或看不见的文本。
- **现象**：静态主按钮翻译后，遥测状态、轨迹 `LIVE/HOLD`、队列待命、历史导出和异常回退仍可能直接渲染英文。
- **判断**：国际化遗漏常发生在“状态文本、空状态、下载文件、fallback”这些非主路径。
- **干预**：建立 locale key 与 display label mapper；逐步替换内部阶段、优先级、导出字段和回退文案；为 HTML 根节点同步语言属性。
- **回归风险**：动态服务端字段不能凭前端猜测翻译；硬编码中文补丁容易造成编码问题。
- **结果**：静态 UI 与本地导出在中英文模式下保持一致，业务动态数据保留后端字段扩展需求。
- **沉淀**：`可复用前端规则`。国际化验收必须包含状态、空态、禁用、通知、导出、回退和 aria 文案。

### 8.2 词典编辑本身也是工程风险

- **用户信号**：Vite 报 `Unterminated string`，后续又出现重复 locale key 的 TypeScript 错误。
- **现象**：Windows PowerShell 默认编码误读 UTF-8 后再写回，中文和引号被破坏；恢复基线后又有部分未提交 key 需要补回。
- **判断**：大行对象词典与平台默认编码组合，极易让文本替换造成不可见损坏。
- **干预**：用显式 UTF-8 读写恢复内容；以构建错误为依据删除重复 key；把词典修改纳入类型构建验证。
- **回归风险**：局部脚本替换会覆盖近期未提交翻译；大补丁难以精确匹配一行词典。
- **结果**：构建恢复，词典键重新可用。
- **沉淀**：`可复用前端规则`。国际化文件必须统一 UTF-8、避免系统默认编码写入，并在每轮文案修改后跑类型构建。

## 阶段 9：工程调试与验证不是最后的形式步骤

### 9.1 语法错误与类型错误必须回到真实根因

- **用户信号**：先后出现 TopNavigation JSX 解析错误、locale 字符串错误、队列优先级 TypeScript 错误。
- **现象**：图标插入到事件处理器、文本替换破坏字符串、展示层使用旧优先级枚举，都会在运行或构建阶段失败。
- **判断**：这些问题不能靠继续叠加补丁解决；必须读取报错位置、比对实际类型和值域、做最小修复并复验。
- **干预**：分别恢复合法 JSX、显式编码修复词典、将队列展示映射从旧 `priority/critical` 改为协议真实 `high/urgent`。
- **回归风险**：修复显示层时误改协议值；为赶进度跳过构建会把错误留到下一次运行。
- **结果**：最终 build、lint、测试和契约校验全部通过。
- **沉淀**：`可复用前端规则`。对编译错误先确认根因和数据模型，再做单变量修复；不要用大范围替换掩盖错误。

### 9.2 交付前需要四类门槛

- **用户信号**：询问代码质量是否足够、Git 是否过度提交、入口描述是否清晰。
- **现象**：前端已能运行，但 README 还是 Vite 模板，包名不具语义，完整源码备份被跟踪进 Git。
- **判断**：可用页面不等于可交接项目。
- **干预**：替换 README；整理运行模式、契约与待办入口；将包名改为 `mcp-2099-agent-console`；从 Git 索引移除 `backups/` 并加入忽略规则；执行 build、lint、test、contract validation。
- **回归风险**：`git rm --cached` 必须只取消跟踪，不删除本机备份；构建产物和日志不能被重新提交。
- **结果**：形成验证通过的检查点提交并推送远端。
- **沉淀**：`可复用前端规则`。前端交付至少检查运行、质量门、入口文档和版本库卫生四个层面。

---

## 阶段 10：停止条件与遗留治理

### 10.1 不把“还能更好”当作继续开发理由

- **用户信号**：明确指出不要无限陷入自我优化，没有必要的内容只做记录。
- **现象**：WebGL 过渡、CSS 整理、性能监控、更多导出格式等都可以继续做，但缺少真实验收或业务触发条件。
- **判断**：没有停止条件的视觉打磨会持续扰动已稳定的页面，并降低后端接入前的可控性。
- **干预**：建立前端待办，将视觉过渡、CSS token 收敛、FPS 监控、丰富导出、真实后端异常体验按触发条件记录。
- **回归风险**：把待办当成隐性承诺；为了完成待办而制造无依据的功能。
- **结果**：当前版本被冻结为 Local Mock 前端基线，后续由真实需求、验收或性能证据驱动。
- **沉淀**：`可复用前端规则`。每轮打磨都应明确“完成定义”和“何时停止”，未触发事项进入待办而非继续改代码。
